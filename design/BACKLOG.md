## 2026-05-16 - mcloop graceful recovery from transient API/infrastructure failures (distinct from genuine task failure)

Why it matters: mcloop's current batch/task retry (observed: 3 attempts, ~50s, then stop with "Remaining: N tasks") treats transient infrastructure errors and genuine task failures identically. During an Anthropic API incident on 2026-05-16, Stage-3.1 burned all 3 attempts on `API Error: 500 Internal server error` in under a minute and halted, forcing a manual mcloop restart once the incident cleared. Transient infrastructure failures are self-identifying in the provider error payload: classify first on structured error signature, e.g. HTTP 5xx or 429 plus provider text like the observed "server-side issue, usually temporary - try again in a moment; if it persists, check status.claude.com". Genuine task failures (agent ran, produced bad code, tests red) do not carry that infrastructure envelope. Design: on signature-positive transient failures, back off (exponential, capped, with max total wait and hard ceiling) and retry without consuming the task's genuine-failure budget; on signature-absent failures, keep today's bounded-retry-then-stop behavior. This is fail-safe: the dangerous direction, endlessly retrying a genuinely broken task, is unlikely because broken tasks do not return 5xx/try-again envelopes; the conservative mistake is tagging real infrastructure failure as genuine, which stops as today and requires manual restart. Also covers the NOTES.md DeepSeek/Sonnet fallback chain, which exhibited the same brittle all-providers-failed behavior under the same incident.

Provenance pointer: Claude conversation of 2026-05-16 (bob ecosystem); observed during the Anthropic API incident that failed bob-tools Stage-3.1. This is a deliberate mcloop design item to be scheduled; related but distinct from the already-filed mcloop/BUGS.md entries (claude-CLI argv-too-long fallback defect; CLAUDE.md reconcile retry-exhaustion) - those are deterministic bugs, this is a resilience design change.

## 2026-05-16 - NOTES.md commit-log summarization - provider options to try (user thinks Haiku is sufficient)

Why it matters: mcloop summarizes the git commit log into English prose for NOTES.md. The current path (DeepSeek primary, Sonnet fallback) fails - but evidence shows DeepSeek itself is healthy (verified via `deepseek -p` from the CLI) and the failures are HTTP 400 (mcloop sending a malformed/oversized request) plus the Sonnet fallback dying with `[Errno 7] Argument list too long`. Strong inference: single root cause is an unbounded git-commit-log payload passed as argv (or oversized body), not the provider - already filed as a deterministic mcloop defect in mcloop/BUGS.md. CONSEQUENCE FOR THIS ITEM: swapping providers alone will NOT fix it; the payload must first be bounded and passed via stdin/file, not argv - that is a precondition for any provider choice. With that precondition met, the provider should be reconsidered because commit-log->prose summarization is low-stakes and does not need a frontier model. User's view: Haiku (a small fast Anthropic model) is likely sufficient for this task. Options to try, recorded for later evaluation: (1) Anthropic Haiku - simplest, but NOTE the resilience caveat: it shares a failure domain with the editor agent (claude_code_agent:opus) which also runs on the Anthropic API, so a single Anthropic incident would take down both the build loop and NOTES.md summarization simultaneously; acceptable for a non-fatal feature but worth a conscious choice. (2) Direct DeepSeek (NOT via OpenRouter) - the user's CLI test proved direct DeepSeek works; the unreliability the user observed is specifically the OpenRouter path, so "keep DeepSeek, drop the OpenRouter hop" may be the smallest correct provider change and keeps NOTES.md summarization in a different failure domain from the Anthropic build loop. (3) A small OpenAI-class model - different failure domain from Anthropic, generally high uptime; more integration work. Explicitly avoid adding OpenRouter as a layer: the user's own evidence implicates it as the unreliable component. Recommended evaluation order once payload-bounding lands: try Haiku for sufficiency-of-quality first (user's hypothesis), but weigh the shared-failure-domain caveat against option (2). This is a design/config decision plus the mandatory payload-bounding fix; it requires a read-only source pass first to establish how mcloop's NOTES.md provider chain and fallback are actually wired (which module, argv vs stdin, is the commit log bounded, where providers are configured) before any change is specified.

Provenance pointer: Claude conversation of 2026-05-16 (bob ecosystem); arises from the same incident/diagnosis as the filed mcloop/BUGS.md NOTES.md-summarization defect. Distinct from that bug entry: the BUGS.md entry is the deterministic payload/argv defect to fix; THIS backlog item is the provider-selection design decision (Haiku vs direct-DeepSeek vs small-OpenAI) to make once the payload fix is in. Surface together when scheduled.
