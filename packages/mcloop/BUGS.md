## Bugs

### Coverage gate demands executed-line proof for structurally un-coverable Python (Protocol/ABC definitions, trivial __init__ shims)

**Symptom**: building writer, batch 2.3 failed the coverage gate repeatedly on `writer/extract/base.py`, `writer/extract/__init__.py`, `writer/extract/plaintext.py` even after an explicit, thorough test (`tests/test_extract_interface.py`) was added to the same batch and run. `markdown.py` passed; the others did not. No amount of test-writing fixes `base.py` because it is a `typing.Protocol` definition (T-000022: "defining an `Extractor` Protocol") whose body is `...` stubs that never execute — coverage can never mark its lines executed. `__init__.py` is a dispatch/re-export shim and `plaintext.py` is a thin module reached only via package dispatch; both hit coverage-mapping gaps. All three had to be waived to proceed.

**Root cause**: `coverage_verify.py:verify_change_covered` exempts non-`.py` inputs but treats every `.py` file as coverage-provable, requiring at least one changed line to be executed by a scoped candidate test. Some Python constructs are structurally un-coverable or coverage-opaque: (a) `typing.Protocol` classes and abstract/`@abstractmethod` interface bodies never execute, so their changed lines can never be "covered"; (b) trivial package `__init__.py` files that only import/re-export/dispatch may not register executed lines under scoped coverage and have no namesake test. These are the Python analog of the already-accepted non-`.py` exemption: a class of changes that need no executed-line proof. The gate lacks that recognition, so it forces a waiver for every interface/shim file in every project — recurring friction and a category error (demanding runtime coverage of code that, by construction, does not run).

**Chosen fix**: extend the coverage gate to recognize a coverage-exempt class of Python files and pass them without executed-line proof (no mapped-test requirement, no forced waiver), the same way non-`.py` inputs are exempt. Detect the class by AST (consistent with `change_class.py`'s AST approach), not by filename: a file is coverage-exempt when it contains only non-executable definitions — e.g. every class in it is a `typing.Protocol` subclass or an ABC whose methods are abstract/`...`/`pass`/docstring-only bodies, and/or the module body is solely imports and re-exports (`__all__`, import-and-rebind) with no executable logic. A file containing any real executable logic stays gated as today. Keep the discipline for genuine behavioral code; only exempt what cannot, by construction, be coverage-proven.
