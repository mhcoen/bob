## Bugs
- [ ] T-000037: The loop has no circuit-breaker for a task that keeps failing the SAME gate across every model tier, so a change a gate cannot validate is retried indefinitely instead of stopping. Observed 2026-07-04/05: bob-tools T-000003 produced a correct, fully-tested fix, but the coverage-verify gate could not map its namesake test to the changed module (see T-000038), so every attempt failed identically. The chain retried opus x3, fell over to codex, then kimi, then looped -- thrashing one task for ~11 hours overnight, exhausting the opus usage cap, and burning kimi tokens, all on an unattended run that survived laptop sleep/wake cycles (the process is only suspended during sleep, not stopped). This is the gate-failure analogue of the T-000027 rate-limit retry burn. Fix: track (task_id, gate-failure signature) across tiers and STOP + notify after N consecutive identical gate failures (suggest N=2 tiers or ~2 full chain passes) instead of cycling forever; the notification should name the unsatisfiable gate and the last diff so a human can waive or fix. Optionally cap total wall-clock or attempts per task across the whole chain. Regression test: a task whose scoped check fails deterministically stops after the bounded number of attempts with a distinct terminal status, not an infinite retry.
- [ ] T-000038: coverage-verify's scoped-candidate discovery fails to map a namesake test to a changed CLI module, forcing a legitimate fix to fail the gate forever. Observed 2026-07-05: with only `bob_tools/planfile/cli.py` changed and `bob_tools/planfile/tests/test_cli.py` adding a `TestErrorFilename` class that imports the module AND drives the full CLI via `main([...])` (executing the changed `_print_parse_error` call sites), `mcloop verify` still reports "gate: unaccounted behavioral change -- bob_tools/planfile/cli.py: no scoped candidate test imports or reaches the changed module". So the transitive-import / namesake discovery in coverage_verify.py does not connect `planfile/tests/test_cli.py` to `planfile/cli.py` despite a direct import and a covering invocation. Investigate the candidate-discovery walk (namesake resolution across the package/tests layout, and whether coverage of code reached only through `main()`/argparse dispatch is being measured) and either fix the mapping or broaden the coverage-proven path to count a test that imports the module and executes its changed hunks. This blind spot is what triggered the T-000037 thrash. Regression test: a change to `planfile/cli.py` plus a `test_cli.py` that imports it and exercises the changed lines passes the scoped gate.
