## Bugs

- [x] T-000028: Batch sessions burn retry attempts on transient rate/session limits and never use chain fallover. `_run_batch` (main.py ~886) returns a plain "failed" on a limit hit, and the caller loop (main.py ~1800) consumes a `batch_attempt` and feeds the limit-noise output tail back as `batch_prior_errors`; the `fallback_model` param is ignored entirely. A few transient limits during a large batch exhaust `max_retries` and `mark_failed` the parent and all children even though nothing failed on the merits. The non-batch path already handles this correctly (`attempt -= 1` on a limit, main.py ~2060) -- mirror it: detect `is_rate_limited`/`is_session_limited` on the batch output, do not consume an attempt, and wait-for-reset/fall over via the same ratelimit machinery. Regression test: canned limit-text batch output does not decrement attempts and does not mark tasks failed. - [ ] T-000030: `_build_command` only applies `_apply_provider_env` when `env is not None` (runner.py ~873), and `_run_session` silently builds its own env when passed none -- so any new call site that forgets to thread one env through both calls runs unrouted on third-party models. This bug class has shipped at least twice (code_edit.py ~404 documents the first fix; the 2026-07-02 audit fixed five more sites, since funneled through `_run_claude_subsession`). Make it impossible rather than merely centralized: have `_build_command` own env construction and return `(cmd, env)` (or add a `_prepare_session` helper used by ALL call sites including code_edit.py), and remove the env-None-skips-routing branch. Note main.py ~504 (`_preflight_chain`) calls `_build_command` purely for its env side effect and discards the command -- evidence the env work is misplaced inside command building. Regression: no runner entry point can produce a session command without provider routing applied.
- [ ] T-000031: Run-summary write failures are silently swallowed: main.py ~1074 wraps `write_run_summary` in `except Exception: return None` and callers ignore the result, so a disk/permission error leaves a stale `.mcloop/runs/latest.json` with no warning -- and anything gating on latest.json (the orchestra smoke-test procedure reads it) then validates against the previous run. Surface the failure to stdout and notify at error level, keeping the run alive. Regression test: a raising summary writer produces a visible warning and does not crash the loop.
- [ ] T-000032: Unreachable graceful-interrupt path: main.py ~2028-2059 wraps `time.sleep(SESSION_LIMIT_POLL)` in try/except KeyboardInterrupt to write an "interrupted" run summary, but the installed SIGINT handler (lifecycle.py ~498-508) calls `os._exit(130)` unconditionally, so KeyboardInterrupt never reaches the main thread and the summary-write branch is dead code. Either remove the dead branch or make the lifecycle handler perform the summary write before exiting -- the mcloop CLAUDE.md smoke-test section already documents that latest.json is NOT updated on SIGINT, and fixing that would be the better outcome. Decide one way; do not leave the dead handler in place.
- [ ] T-000033: Batch selective rollback parses `git diff --name-only` / `git ls-files --others` output with `.splitlines()` and feeds the paths to checkout/unlink/rmtree (main.py ~1013-1037): `core.quotepath` escaping of non-ASCII names produces quoted paths that match nothing, embedded newlines break the split, and `shutil.rmtree` on a symlink-to-directory raises -- so a failed batch may not fully restore the pre-batch tree before retry. Use `-z` NUL-delimited output for both commands and guard the symlink case. Regression test: rollback restores a file with a non-ASCII name.

- [ ] T-000036: Installed hooks drift silently from the repo copies: ~/.claude/hooks/telegram-permission-hook.py dated 2026-05-29 (525 lines) was missing the entire McLoop test-routing policy that the repo copy (722 lines) ships, so on 2026-07-03 an editor session ran free-form pytest through a Telegram ask instead of the instant mcloop-verify deny, and the whole test-discipline gate was inactive without any signal. Regression test: a stale installed hook triggers the warning path.
- [ ] T-000034: Formatting discipline varies by editor model: the codex/gpt-5.5 fallover tier ships unformatted code that Claude-family editors would have formatted, so a model swap mid-phase changes the artifact quality. Observed 2026-06-11 (writer, task 20.5): fable failed 3 attempts, the chain fell over kimi/deepseek (402) to codex/gpt-5.5, which completed the task with a test file that failed the phase boundary's `ruff format --check` -- the task-level acceptance (pytest) passed, so the defect surfaced only at the boundary, stopping the run a phase late and requiring a hand fix. Formatting must not depend on which model held the editor seat. Fix: run the project's formatter deterministically in the loop after every editor session and before the acceptance check (e.g. `ruff format` on the changed files when the project configures ruff), the same way commits are mcloop's job and not the model's -- format-on-exit is scaffolding work, not a model habit to hope for. Regression test: a stubbed editor session that writes a deliberately misformatted file still yields a task commit that passes `ruff format --check`.
