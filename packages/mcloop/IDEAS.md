# Ideas

A flat scratchpad for ideas not yet ready to become PLAN.md tasks.
Add anything here — feature sketches, half-baked thoughts, things to
explore later. McLoop does not read or modify this file during runs.
Use `mcloop idea "some text"` to append from the command line.

- [2026-04-10] Branch/worktree isolation for normal mcloop runs. Investigation mode already does this. The main run_loop still mutates the live working tree, which means a mid-run crash, kill, or botched rollback can leave the real working directory dirty. Per-run worktrees would shrink the blast radius, enable parallel runs against the same project, and make interrupted-run recovery cleaner. Open design questions: venv handling per worktree (share parent, recreate, or use a faster dependency cache like uv), auto-wrap interaction (the wrap markers live in source files and would be lost if a worktree is discarded), and whether this should be opt-in via --worktree or eventually become the default. Likely opt-in first for high-risk runs (overnight, bug-only mode, long stages) and leave quick interactive runs in-place.

- [2026-04-13] Built-in design-review consulting (multi-AI critique) before and during long runs. Empirically, having Codex review duplo's redesign documents before mcloop ran them caught several substantive contradictions and one significant safety issue (the prompt-injection invariant in format_spec_for_prompt) that would have been encoded into code as bugs. Mcloop's own audit rounds during the Phase 3 run caught more (13 bugs across two rounds, plus the dead-code cleanup of runner.py). The pattern is consistent: external review before code lands prevents bugs more cheaply than fixing them after. Worth considering making this a first-class feature: (a) `mcloop consult <doc-or-plan> --reviewer codex` runs a structured review pass against design docs or PLAN.md before starting work, surfaces contradictions, missing pieces, scope/sequencing errors, with an aggressive filter for significance; (b) make audit rounds (already implemented) configurable to run a different reviewer model than the implementer (e.g., implement with claude, audit with codex, or vice versa) for cross-model bug-finding; (c) add a third-model option — DeepSeek (V4 or successor) is interesting because it's much cheaper and has a different training distribution, so it may catch things claude/codex both miss and at low cost can be run liberally. Open design questions: how to invoke each reviewer (claude CLI, codex CLI, deepseek API), how to structure the review prompt to filter for significance (the empirical lesson is that without explicit "do NOT raise" lists, reviewers surface stylistic noise that drowns out real issues), where review output goes (.mcloop/reviews/ alongside errors.json?), and whether duplo or mcloop is the right home (probably mcloop — it owns the run loop and audit infrastructure, and duplo could call into it). Adjacent: revisit whether mcloop's existing audit rounds should default to a different model than the one running tasks, since same-model audit has limited bug-finding power compared to cross-model audit.
