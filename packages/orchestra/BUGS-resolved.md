## Resolved Bugs

- [x] The `codex` model identifier resolves to `gpt-5-codex`, which ChatGPT-account Codex rejects with `400 invalid_request_error: The 'gpt-5-codex' model is not supported when using Codex with a ChatGPT account` -- wrong default ecosystem-wide. Observed in three places: duplo plan_author reviewer leaf (`{"model": "codex"}`) errored its review state (orchestra run ed723f36e53b on /Users/mhcoen/proj/writer carries the 400 verbatim); mcloop chain tier 2 (codex/gpt-5-codex) fails preflight and is skipped every run; any run_role/workflow leaf using the `codex` identifier. Root cause: `orchestra/registry/registry.py` BUILTIN_MODEL_IDENTIFIERS["codex"] is ModelIdentifier(name="codex", adapter="codex_text", model="gpt-5-codex"), but the account's Codex CLI serves gpt-5.5 (confirmed: shell `codex` reports model: gpt-5.5) -- one wrong constant, not per-project misconfiguration; patching individual project configs masks it one site at a time. Fix: change the `codex` identifier's model to the model the account serves (gpt-5.5) at the source in BUILTIN_MODEL_IDENTIFIERS, then audit every package for independent gpt-5-codex string literals that bypass the identifier table (mcloop's chain-tier config is one -- see mcloop/BUGS.md) and fix those too. Decide whether gpt-5-codex remains a selectable identifier at all (for accounts that DO serve it) or is removed; if kept it must not be the value behind the bare `codex` identifier everything defaults to.
- [x] The progress reporter labels states with the role binding's configured model, ignoring the invocation-options model override -- an operator watching a run sees the wrong model in every progress line whenever an override is in effect. Observed 2026-06-10 on an mcloop-driven code_edit run: mcloop's tier-1 chain passed model="fable" as the override and the live subprocess genuinely ran `claude -p ... --model fable` (verified against mcloop's .mcloop/active-pid record), yet every progress line read `editor (claude_code_agent:opus)` for the whole 17-minute session -- the editor role's configured binding at the time -- and the operator reasonably concluded the wrong model was coding. The label is correct only when the binding happens to coincide with the override. Root cause, two independent paths: the executor folds invocation_options into backing_options at invoke time so the ADAPTER builds the command with the override model, but `_wrap_progress_callback` (`orchestra/api/bindings.py`) enriches every ProgressEvent via `_resolve(role)`, which returns `(binding.adapter, binding.model)` straight from the role binding, and `_format_backing` (`orchestra/progress.py`) renders that stale value; nothing in the progress path consults the override. Fix: resolve the EFFECTIVE model (the binding overridden by invocation_options) before emitting progress events, so the label always names the model the adapter actually invokes. Regression test: run a workflow with a stub adapter, a role bound to model A, and invocation_options {"model": "B"}; assert the captured ProgressEvent.model and the format_event rendering say B, not A. A monitoring line that misreports which model is executing defeats the purpose of having one.
- [x] T-000001: `run_workflow` leaks the SQLite ArtifactStore on any executor/summary exception: api/dispatch.py ~256 opens the store but `store.close()` runs only on the success paths (~320/~325); the try/finally at ~300-307 closes the log, not the store, so each errored run in a long-lived REPL/server process leaks a connection/fd until exhaustion.
