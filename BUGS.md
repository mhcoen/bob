- [x] T-000027: Rate-limit fallover is wired ONLY into the main task-execution loop; five other session-spawning paths fail open on 429/session-limit. When the Opus 5-hour limit is hit during audit / bug-verify / bug-fix / post-fix-review / diagnostic (and batch sessions mid-run), mcloop treats the rate-limit rejection as an ordinary session failure and aborts, with no fallover to an alternate CLI, no wait-for-reset, and no user notification. Observed: a 5-hour-limit 429 during the audit phase produced `audit: session exited with code 1, skipping fix` (audit.py ~164-169), which propagated to a terminal "Audit failed" and blocked phase completion, even though all task work had succeeded. Root cause: run_audit/run_bug_verify/run_bug_fix/run_post_fix_review/run_diagnostic (runner.py ~1209-1351) all go through `_run_session` returning `RunResult(success=returncode==0)`, and the consumers branch only on `.success`/`.exit_code` and never inspect `.output` for the rate-limit/session-limit text that ratelimit.py's `is_rate_limited`/`is_session_limited` would match. Fix: wrap these five paths' results in the same `is_session_limited`/`is_rate_limited` -> `mark_limited` -> `wait_for_reset`/`get_available_cli` handling the task loop already uses (main.py ~1775/1841), so a transient limit during a sub-session waits for reset / falls over rather than aborting the run. Two secondary fixes belong with this: (a) `_run_batch` picks an available CLI before running but never re-checks the batch session OUTPUT for a limit hit mid-batch, so it also fails open — add the same post-result check; (b) limit events currently surface only to Telegram via `notify()` (notify.py ~76-87), never to mcloop stdout except for the task loop's separate explicit `print()` calls — surface limit/wait/fallover events to stdout for ALL paths so the watching user sees them. Note the existing fallover default reach is `claude`->`codex` only (ratelimit.py `ALL_CLIS`); broadening to kimi/deepseek tiers is the separate model-resumption backlog item in NOTES.md, not part of this fix. This bug class was shipped untested — add regression tests that feed a canned 429/session-limit stream-json output through each of the five sub-session paths and assert the limit is detected and handled (wait/fallover) rather than treated as failure. <!-- completed_at: 2026-05-29T04:22:43Z -->
